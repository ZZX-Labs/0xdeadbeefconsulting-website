Documentation Standards

Clear, Auditable, and Maintainable

Technical documentation is written to support implementation, review, operation, troubleshooting, security analysis, and future maintenance.

Versioned

Documents should identify the applicable project, software version, hardware revision, publication date, and revision history where relevant.

Reproducible

Installation steps, commands, dependencies, configuration examples, and expected outcomes should be specific enough to reproduce reliably.

Security-Aware

Documentation should distinguish examples from production configurations and avoid publishing secrets, credentials, private keys, or unnecessary sensitive data.

Explicit

Assumptions, prerequisites, permissions, limitations, failure conditions, and unsupported configurations should be stated directly.

Portable

Plain text, Markdown, HTML, PDF, and other durable formats are preferred over proprietary or inaccessible documentation systems.

Reviewable

Public documents should be structured for peer review, issue reporting, correction, and controlled revision.

Publication Status

Public, Limited, or Private

Not all technical material can be published. Documentation is categorized according to ownership, confidentiality, safety, legal requirements, and client authorization.

Public

Open documentation, source references, general manuals, research papers, public reports, and authorized disclosures available without restricted access.

Limited Distribution

Materials shared only with verified clients, collaborators, maintainers, or authorized reviewers under defined handling terms.

Private

Client-confidential records, security-sensitive findings, credentials, private architecture, evidence, and other material not authorized for publication.

Publication limitation: The absence of a public report, source file, advisory, or technical record does not imply that no related work exists. Some material may remain private by contract, law, safety requirement, or professional obligation.

Corrections & Contributions

Report Documentation Problems

Documentation quality depends upon correction, review, version control, and responsible feedback.

Useful Reports Include

  • The affected document or page URL.
  • The section, heading, command, or line involved.
  • The relevant project and version.
  • The operating system or environment.
  • The observed problem.
  • The expected result.
  • A proposed correction, if available.

Security-Sensitive Errors

Do not publish security-sensitive documentation defects in public issue trackers before coordinated review.

Report problems involving authentication, authorization, cryptography, secret handling, exploitability, unsafe defaults, or operational exposure through the secure contact process.
